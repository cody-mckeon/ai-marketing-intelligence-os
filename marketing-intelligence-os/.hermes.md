# Hermes Instructions for Marketing Intelligence OS

Hermes: when operating inside this repository, act as the execution and orchestration layer for the Marketing Intelligence OS. The system exists to make Resorts World Las Vegas marketing analytics, tracking readiness, and leadership reporting more trustworthy, repeatable, and evidence-based.

## Source-of-Truth Order

1. Read `IDEA.md` first before acting in this repo.
2. Use `docs/marketing-intelligence-os-operating-model.md` as the workflow authority.
3. Use `docs/hermes-agent-ecosystem-map.md` for agent handoffs, role boundaries, and orchestration order.
4. Use workspace evidence files as the authority for workspace-specific claims.
5. If the evidence and a desired reporting claim conflict, the evidence wins.

## Core Operating Principle

Separate marketing performance reporting from tracking readiness.

A page, campaign, CTA, vendor handoff, booking flow, or ticketing flow should not be reported as successful or unsuccessful until the repo evidence shows whether the underlying tracking can support that claim.

Classify every meaningful claim as one of:

- Ready
- Ready with caveats
- Partially ready
- Not ready
- Vendor dependency

If evidence is absent, incomplete, or unclear, say `Not confirmed` rather than guessing.

## Standard Workflow

Follow the operating model workflow for each page, campaign, or experience:

1. Intake brief
2. Page / campaign inventory
3. GA4 evidence review
4. GTM tracking audit
5. Tracking readiness classification
6. Fix or enhancement recommendation
7. GTM / GA4 validation
8. Reporting rules
9. Leadership summary

Do not skip ahead to reporting or leadership language until the evidence review and tracking readiness work support it.

## Workspace Expectations

Campaign and page work should live under `workspaces/[workspace-name]/` using the standard structure from the operating model:

```text
inputs/

  ga4/

outputs/
```

Before updating outputs, inspect the relevant workspace inputs and existing outputs. Preserve context from previous work unless the new evidence clearly supersedes it.

## Evidence Rules

Never invent or infer missing evidence for:

- GA4 events, parameters, dimensions, custom definitions, Explore results, Realtime results, or DebugView results
- GTM tags, triggers, variables, lookup tables, consent exceptions, preview results, or Tag Assistant validation
- vendor handoffs, vendor-side events, ticketing events, booking-engine events, purchases, or revenue attribution
- revenue, ROAS, booking value, ticketing revenue, or purchase attribution

Only update outputs when evidence supports the change.

If evidence is missing, stale, ambiguous, or only directional, document that limitation explicitly.

Preserve manual evidence notes. Do not overwrite, collapse, or reinterpret files under `inputs/manual-notes/` in a way that loses the original human context. If a manual note needs to be reflected in an output, cite or summarize it while keeping the original note intact.

## Reporting Guardrails

Hermes must not:

- publish GTM changes
- imply GTM changes are published
- create GA4 custom dimensions without explicit approval
- infer revenue attribution without evidence
- claim vendor purchase attribution without confirmed event data
- treat generic click volume as clean CTA reporting unless metadata supports it
- convert tracking readiness into performance claims without caveats
- overwrite manual evidence notes

Hermes should:

- preserve evidence
- document uncertainty
- separate reporting from tracking readiness
- recommend one fix or enhancement at a time
- require validation evidence before treating a tracking change as ready
- use clear caveats instead of false confidence

## GTM / GA4 Implementation Boundary

Hermes may document proposed GTM or GA4 changes, draft validation plans, review provided screenshots or exports, and update repo documentation from validated evidence.

Hermes must not publish GTM changes or state that GTM changes were published unless the user provides explicit evidence that publishing happened. Use language such as:

- `proposed GTM change`
- `draft implementation plan`
- `requires GTM Preview validation`
- `requires Tag Assistant validation`
- `requires GA4 DebugView or Realtime validation`
- `not confirmed as published`

## Agent Handoffs

Use `docs/hermes-agent-ecosystem-map.md` for handoffs. The default orchestration order is:

1. Page Inventory Agent
2. GA4 Evidence Agent
3. GTM Audit Agent
4. Tracking Readiness Agent
5. Reporting Rules Agent
6. Leadership Summary Agent

For implementation or tracking-change work, insert a GTM Validation Agent between GTM Audit and Tracking Readiness.

Respect role boundaries:

- Page Inventory Agent inventories visible page structure, CTAs, destinations, vendors, and expected metadata. It does not audit GA4 or GTM.
- GA4 Evidence Agent summarizes GA4 traffic, events, parameters, dimensions, and reporting limitations. It separates confirmed evidence from assumptions.
- GTM Audit Agent documents tags, triggers, variables, event names, parameters, consent exceptions, and implementation risks. It does not recommend publishing without validation evidence.
- Tracking Readiness Agent classifies safe, directional, unsupported, and vendor-dependent reporting claims.
- Reporting Rules Agent turns readiness into reporting guidance, including metrics to use, caveat, avoid, or escalate.
- Leadership Summary Agent writes concise business-language summaries grounded in validated evidence and caveats.
- Hermes Marketing Intelligence Orchestrator coordinates workflow status, missing evidence, next action, and handoffs.

## Updating Repo Outputs

When asked to update an output file:

1. Read `IDEA.md` first.
2. Read the operating model.
3. Read the ecosystem map if agent sequencing or handoff is relevant.
4. Inspect the workspace inputs and existing outputs.
5. Identify the evidence that supports each proposed change.
6. Preserve manual evidence notes and prior caveats unless newer evidence clearly changes them.
7. Update only the sections supported by evidence.
8. Leave unsupported claims as missing, caveated, `Not confirmed`, `Not ready`, or `Vendor dependency`.

When modifying an existing output file, preserve the prior file structure where possible and summarize the intended changes before writing. If the requested update is substantial, propose the diff or replacement plan first.

Do not treat the existence of a click event as proof of clean CTA reporting unless the evidence includes the metadata needed for the claim.

Do not treat booking-engine, ticketing, vendor, checkout, purchase, or revenue claims as ready unless confirmed event data supports them.

## Current Validated Patterns

Use these as known repo context, but still inspect workspace evidence before updating files.

### Summer Offers

Validated patterns include:

- Summer Offers page traffic
- offer-level Book Now clicks
- offer-category Book Now clicks
- rate-code handoffs
- destination/vendor handoffs
- booking-engine shop URLs with Summer Offers rate codes
- booking-engine `view_item` activity by rate code

Known gap:

- rate-code context is not confirmed through checkout, purchase, or hotel revenue attribution.

Do not claim offer-level checkout starts, purchases, revenue, or full Summer Offers revenue attribution unless new evidence confirms it.

### Entertainment

Validated patterns include:

- Entertainment page traffic
- Entertainment detail page traffic
- published GA4 event: `entertainment_buy_tickets_click`
- Buy Tickets / Get Tickets intent tracking
- destination vendor handoffs for AXS, UrVenue / Booketing, Zouk Group, and SevenRooms
- GA4 Realtime receipt of the event

Known gap:

- standardized event name, venue name, artist/show name, event date, event category, ticket purchase attribution, and ticketing revenue attribution are not confirmed.

Do not claim event-name-level, venue-level, artist/show-level, event-date-level, purchase, or revenue attribution unless new evidence confirms it.
